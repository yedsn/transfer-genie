# 自动更新方案

本文档用于梳理 Transfer Genie 的自动更新设计方案，目标是：

- 用户在设置页勾选“自动更新”后，应用在主窗口显示后于后台静默检查新版本
- 若检测到新版本，弹出确认框询问是否更新
- 用户确认后，执行下载、安装，并在完成后提示是否立即重启应用

本文档聚焦桌面端 Tauri 应用，不包含移动端方案。

## 一、目标与范围

### 目标

- 提供一个用户可控的自动更新开关
- 在不打扰用户正常启动流程的前提下，自动检查更新
- 发现新版本时给出明确且统一风格的确认交互
- 更新失败时不影响应用主功能
- 为后续“手动检查更新”“跳过此版本”“稳定版 / 测试版渠道”预留扩展空间

### 非目标

- 本次不设计灰度发布
- 本次不设计多更新通道切换
- 本次不实现强制更新
- 本次不覆盖移动端或 Linux 特殊发行渠道差异

## 二、当前项目现状

从当前代码结构看，自动更新能力适合接入到现有设置、窗口事件和前端弹框体系中。

### 已有能力

- 设置统一由 Rust `Settings` 管理，前端通过 `get_settings` / `save_settings` 读写
- 设置页已有“系统设置”分组，适合新增“自动更新”开关
- 主窗口显示时，Rust 会发送 `trigger-show` 事件给前端
- 前端已具备统一的信息弹框、确认弹框能力
- 应用本身已有托盘和窗口显示 / 隐藏逻辑，适合在“窗口真正显示后”再触发检查

### 现有关键落点

- 设置结构：`src/types.rs`
- 设置读取 / 保存：`src/main.rs`
- 主窗口显示事件：`src/main.rs`
- 设置页 UI：`frontend/index.html`
- 设置页逻辑与弹框逻辑：`frontend/main.js`

## 三、推荐总体方案

推荐采用：

- Rust 侧接入 `tauri-plugin-updater`
- 前端不直接操作 updater 插件，而是调用自定义 Tauri command
- 前端负责交互与展示
- Rust 负责检查更新、下载、安装、重启

这样做的原因：

- 当前前端是静态 HTML / JS，不是完整的 npm 工程；Rust 包一层更容易落地
- 更新逻辑、错误处理、节流、日志更适合收口在 Rust
- 前端可以继续复用现有对话框风格，避免系统原生弹窗风格不一致

## 四、用户体验流程

### 自动检查流程

1. 用户在设置页勾选“自动更新”
2. 应用主窗口显示
3. 前端收到 `trigger-show`
4. 延迟 2 到 5 秒后发起静默检查
5. 若没有更新，静默结束
6. 若有更新，弹出确认框
7. 用户点击“立即更新”后开始下载和安装
8. 安装完成后提示“立即重启”或“稍后重启”

### 交互原则

- 自动检查失败：不弹错误框，最多记日志或轻提示
- 手动检查失败：明确提示失败原因
- 同一版本在同一次启动中不要反复弹框
- 检测阶段尽量静默，不打断主界面可用性

## 五、设置设计

### 第一阶段建议字段

建议先增加一个最小字段：

```json
{
  "auto_update_enabled": false
}
```

对应 Rust `Settings` 增加：

```rust
pub auto_update_enabled: bool
```

### 第二阶段可扩展字段

如果后续要优化体验，可以增加：

```json
{
  "auto_update_enabled": true,
  "skipped_update_version": "0.1.2",
  "last_update_check_at_ms": 1710000000000
}
```

字段含义：

- `auto_update_enabled`：是否允许自动检测新版本
- `skipped_update_version`：用户点击“暂不更新”的版本号，避免同版本反复提示
- `last_update_check_at_ms`：上次检查时间，用于做频率限制

### 建议策略

第一版只落 `auto_update_enabled` 即可，降低改动范围。

## 六、前端设计

### 设置页

在“系统设置”分组中增加：

- `自动更新` 开关
- 可选：`手动检查更新` 按钮

推荐文案：

- 开关标题：`自动更新`
- 开关说明：`打开后会在窗口显示后后台检查新版本`
- 按钮文案：`检查更新`

### 前端状态建议

前端增加以下运行时状态：

- `hasAutoUpdateCheckedThisSession`
- `isAutoUpdateChecking`
- `updatePromptedVersion`

用途：

- 防止一次会话内重复检查
- 防止并发发起多次检查
- 防止同一版本重复弹窗

### 自动检查触发点

推荐挂在现有 `trigger-show` 事件处理链路中，而不是页面首次脚本执行时立即检查。

原因：

- 应用刚启动时主线程和页面渲染更忙
- 主窗口真正显示后再检查，用户体验更自然
- 托盘恢复窗口时也能复用同一事件模型

### 弹框设计

检测到可更新版本后，弹出确认框展示：

- 当前版本
- 最新版本
- 发布时间
- 更新说明

确认框按钮建议：

- 主按钮：`立即更新`
- 次按钮：`稍后再说`

安装完成后再弹一个提示框：

- 标题：`更新已安装`
- 按钮：`立即重启` / `稍后重启`

## 七、后端设计

### 推荐 Tauri command

建议新增以下命令：

- `check_app_update`
- `download_and_install_update`
- `restart_app`

### `check_app_update`

职责：

- 调用 updater 检查服务端是否存在新版本
- 返回结构化结果给前端

建议返回结构：

```json
{
  "available": true,
  "currentVersion": "0.1.0",
  "version": "0.1.1",
  "notes": "修复若干问题",
  "pubDate": "2026-04-20T12:00:00Z"
}
```

若无更新：

```json
{
  "available": false,
  "currentVersion": "0.1.0"
}
```

### `download_and_install_update`

职责：

- 执行下载和安装
- 将进度事件推送给前端
- 失败时返回明确错误

建议通过事件推送：

- `update-download-started`
- `update-download-progress`
- `update-install-finished`
- `update-install-failed`

### `restart_app`

职责：

- 在安装完成后，用户确认重启时执行应用重启

是否直接使用 updater 自带重启能力或 process 插件重启能力，可在实现阶段按 Tauri 版本兼容性确认。

## 八、运行时状态机

```text
主窗口显示
  -> 自动更新开关关闭
      -> 结束
  -> 本次会话已检查
      -> 结束
  -> 当前已有检查任务
      -> 结束
  -> 延迟 2~5 秒
  -> 检查更新
      -> 无更新
          -> 标记本次会话已检查，结束
      -> 有更新
          -> 弹确认框
              -> 用户取消
                  -> 标记本次会话已提示，结束
              -> 用户确认
                  -> 下载并安装
                      -> 失败：提示失败
                      -> 成功：提示是否立即重启
```

## 九、节流与防重复策略

### 最小可用策略

- 每次应用启动只自动检查一次

### 建议增强策略

在“每次启动只检查一次”的基础上，再增加：

- 距离上次检查不足 6 小时则不自动检查
- 用户本次已经取消同版本更新，不重复提示

### 为什么要做节流

- 避免窗口隐藏 / 显示造成重复网络请求
- 避免更新服务短时间被重复命中
- 避免用户被同一个版本频繁打扰

## 十、发布链路要求

自动更新不仅是应用内逻辑，还依赖发布产物与更新服务。

### 必需能力

- 生成签名密钥
- 构建时为更新产物签名
- 生成 updater 所需的更新产物
- 上传更新产物、签名文件、更新元数据
- 应用配置中写入公钥和更新地址

### 当前项目需要补齐的点

- `tauri.conf.json` 中补 updater 配置
- 打包脚本中加入 updater 产物生成
- 发布流程中上传 `latest.json` 和对应更新包 / 签名文件
- Windows 发布资产改为 installer 路线，而不是仅依赖本地替换 `exe`

### 推荐的 GitHub Releases 发布形态

如果更新信息和更新包都放在 GitHub，推荐直接使用 GitHub Releases 作为统一托管位置。

每次发布一个版本时，Release assets 至少应包含：

- Windows 更新包
- Windows 更新包对应的 `.sig`
- macOS 更新包
- macOS 更新包对应的 `.sig`
- `latest.json`

其中：

- 面向自动更新的 Windows 资产应是标准 installer 对应的 updater 产物
- 面向自动更新的 macOS 资产应是 `.app.tar.gz` 这一类 updater 产物，而不是仅上传 `.dmg`
- `.sig` 文件需要和对应更新包严格匹配，每次重新构建后都应重新生成

### 推荐的 GitHub endpoint

若采用静态 JSON，客户端可直接请求 GitHub Release 中的固定下载地址：

```text
https://github.com/<owner>/<repo>/releases/latest/download/latest.json
```

优点：

- 客户端配置简单
- 只要最新 Release 正确发布，客户端就能自动跟随
- 不需要额外搭建更新服务器

注意：

- 若仓库是私有仓库，默认下载会涉及鉴权，不适合作为第一版方案
- 第一版建议使用公开仓库或公开的 Release 资产

### `tauri.conf.json` 推荐配置

使用 GitHub Releases 时，配置可参考：

```json
{
  "bundle": {
    "createUpdaterArtifacts": true
  },
  "plugins": {
    "updater": {
      "pubkey": "CONTENT FROM PUBLICKEY.PEM",
      "endpoints": [
        "https://github.com/<owner>/<repo>/releases/latest/download/latest.json"
      ]
    }
  }
}
```

说明：

- `createUpdaterArtifacts: true` 用于生成 updater 所需资产
- `pubkey` 必须是公钥内容本身，不是文件路径
- `endpoints` 可以配置多个，只有前一个返回非 2xx 时才会继续尝试下一个
- 若后续需要同时保留备用源，可以再补第二个 endpoint

### Windows 安装模式

若后续要优化 Windows 安装体验，可在 updater 配置中增加：

```json
{
  "plugins": {
    "updater": {
      "windows": {
        "installMode": "passive"
      }
    }
  }
}
```

当前更建议先使用默认模式或 `passive`，不建议第一版直接上 `quiet`。

## 十一、更新源设计

### 方案 A：静态 JSON

适合第一版快速落地。

特点：

- 发布时生成更新元数据文件
- 客户端直接请求固定 URL
- 可以托管到 GitHub Releases、对象存储或静态站点

优点：

- 简单
- 成本低
- 易于调试

缺点：

- 后续做渠道、灰度、分平台策略时灵活性一般

### 方案 B：动态更新接口

服务端根据平台、架构、当前版本动态返回更新信息。

优点：

- 适合未来扩展稳定版 / 测试版
- 可以加灰度和白名单

缺点：

- 实现和运维成本更高

### 推荐

第一版使用静态 JSON。

### 本项目推荐落地：GitHub Releases + 静态 JSON

结合当前项目情况，推荐使用：

- GitHub Releases 存放更新包
- GitHub Releases 同时存放 `.sig`
- GitHub Releases 同时存放 `latest.json`
- Tauri updater 从 `releases/latest/download/latest.json` 拉取最新信息

这是当前成本最低、最容易验证的实现路径。

### `latest.json` 建议结构

根据 Tauri updater 静态 JSON 约定，建议结构如下：

```json
{
  "version": "v0.1.1",
  "notes": "修复自动刷新和设置页若干问题",
  "pub_date": "2026-04-20T12:00:00Z",
  "platforms": {
    "windows-x86_64": {
      "signature": "SIG_FILE_CONTENT",
      "url": "https://github.com/<owner>/<repo>/releases/download/v0.1.1/transfer-genie-setup.exe"
    },
    "darwin-aarch64": {
      "signature": "SIG_FILE_CONTENT",
      "url": "https://github.com/<owner>/<repo>/releases/download/v0.1.1/Transfer%20Genie.app.tar.gz"
    },
    "darwin-x86_64": {
      "signature": "SIG_FILE_CONTENT",
      "url": "https://github.com/<owner>/<repo>/releases/download/v0.1.1/Transfer%20Genie.app.tar.gz"
    }
  }
}
```

注意事项：

- `version`、`platforms.[target].url`、`platforms.[target].signature` 是必填
- `signature` 必须填写 `.sig` 文件内容本身，不是 `.sig` 文件链接
- `pub_date` 若填写，必须使用 RFC 3339 格式
- `platforms` 的 key 默认使用 `OS-ARCH` 形式，例如 `windows-x86_64`、`darwin-aarch64`
- Tauri 会先校验整个 JSON 文件，再判断版本，因此文件中每个平台项都必须完整有效

### GitHub 发布动作建议

推荐将发布过程固化为脚本或 GitHub Actions：

1. 设置签名私钥环境变量
2. 执行 `cargo tauri build`
3. 收集 updater 产物与 `.sig`
4. 生成 `latest.json`
5. 创建 / 更新 GitHub Release
6. 上传所有 release assets

如果后续接 CI，建议优先采用 GitHub Actions 自动发布，避免手工上传导致漏传签名或 `latest.json` 内容不一致。

## 十二、平台注意事项

### Windows

- 推荐基于标准 installer 更新
- 不建议继续以“直接覆盖运行中的 exe”为自动更新主路线
- 若后续支持静默安装，需要进一步评估 install mode 和 UAC 体验

### macOS

- 需要注意签名、公证和更新包格式
- 自动更新涉及 `.app.tar.gz` 等 updater 产物，而不只是 `.dmg`

### GitHub 托管下的平台补充说明

- GitHub Releases 可以同时放手动下载资产和自动更新资产
- 面向用户下载的 `.dmg` 可以保留
- 面向 updater 的 `.app.tar.gz`、installer、`.sig` 需要单独完整上传
- 若以后要支持 Intel Mac 和 Apple Silicon，建议分别确认产物命名与 `platforms` 键是否匹配

## 十三、错误处理策略

### 自动检查阶段

- 网络失败：静默记录
- 服务端响应非法：静默记录
- 签名 / 版本信息异常：静默记录

### 用户确认更新后

- 下载失败：弹框提示“下载失败”
- 安装失败：弹框提示“安装失败”
- 重启失败：弹框提示“请手动重启应用完成更新”

### 日志建议

建议日志统一带前缀：

- `updater_check_started`
- `updater_check_succeeded`
- `updater_check_failed`
- `updater_install_started`
- `updater_install_succeeded`
- `updater_install_failed`

## 十四、推荐实施顺序

### 阶段 1：最小闭环

- 设置页增加“自动更新”开关
- Rust 接入 updater 插件
- 增加“手动检查更新”
- 能检查到更新并弹确认框

### 阶段 2：自动检查

- 在 `trigger-show` 后延迟触发自动检查
- 增加“本次启动仅检查一次”
- 自动检查失败保持静默

### 阶段 3：安装与重启

- 下载并安装更新
- 展示进度
- 安装后提示立即重启

### 阶段 4：体验优化

- 跳过此版本
- 限制检查频率
- 显示更新时间和更新摘要

## 十五、测试建议

### 功能测试

- 自动更新关闭时，显示窗口不检查
- 自动更新打开时，显示窗口触发检查
- 无更新时不打扰用户
- 有更新时只弹一次确认框
- 用户取消后本次会话不重复弹
- 用户确认后可以完成下载、安装、重启

### 异常测试

- 更新地址不可达
- 更新元数据损坏
- 签名校验失败
- 下载中断
- 安装失败
- 用户点击稍后重启

### 回归测试

- 不影响现有启动流程
- 不影响托盘显示 / 隐藏
- 不影响设置保存
- 不影响当前消息收发、Telegram Bridge、HTTP API 等主功能

## 十六、建议的第一版实现结论

综合当前项目结构，建议第一版采用以下最小可行方案：

- 在设置页增加 `自动更新` 开关
- Rust 侧接入 `tauri-plugin-updater`
- 前端通过自定义 command 调用更新检查与安装
- 挂在 `trigger-show` 事件上，延迟几秒后静默检查
- 一次启动只检查一次
- 有更新时弹框确认
- 安装完成后让用户选择是否立即重启

这个方案改动面适中、路径清晰，且与当前项目的设置模型、窗口事件模型和弹框体系兼容度最高。
