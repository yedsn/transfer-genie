## Context
现有标记系统已经会在历史记录和同步到本地的消息中持久化 `marked`、`marked_tag_ids` 和 `marked_pinned`，但发送命令创建的新消息始终默认为未标记状态。首页输入区目前也没有发送前元数据控制能力，更没有把标签目录改动与一次发送绑定在一起的暂存模型。

## Goals / Non-Goals
- Goals:
- 允许用户在首页输入区发送前直接决定本条消息是否标记。
- 允许用户在发送前通过弹出层选择标签、新增标签、删除标签。
- 让发送出的消息在写入历史记录和本地索引时就已经带有已标记状态和标签信息。
- 让本次发送中的标签目录改动只在发送成功时才真正保存。
- 保持现有发送后再标记的流程不变。
- Non-Goals:
- 不支持发送时直接设置置顶状态。
- 不改动发送流程以外的标记页管理行为。

## Decisions
- Decision: 首页输入区使用独立“标记”勾选作为本条消息是否进入已标记状态的主开关，标签选择只是附加元数据。
- Alternatives considered: 仅在选择标签时自动视为已标记。拒绝原因是无法覆盖“只想标记、不想分配标签”的场景。
- Decision: 标签选择、新增、删除都先保存在输入区暂存态中，发送成功时再统一提交。
- Alternatives considered: 用户在弹出层里一操作就立刻落库。拒绝原因是这会让“准备发送但最终未发送”的尝试污染标签目录。
- Decision: 发送接口接收“是否标记”“最终选中标签 ID”“本次待新增标签”“本次待删除标签”四类发送时上下文。
- Alternatives considered: 发送成功后再额外调用一次 `mark_message` 和若干标签管理命令。拒绝原因是需要多次往返，且会暴露中间状态。
- Decision: 发送成功后清空输入区中的标记和标签暂存状态；发送失败时保留。
- Alternatives considered: 发送成功后保留上一次状态。拒绝原因是更容易误带到下一条消息。

## Risks / Trade-offs
- 输入区的 UI 密度会略有上升。
- 发送接口会明显变宽，因为需要增加标记状态、标签选择和标签目录暂存变更参数。
- “发送后统一保存”会让标签管理逻辑比现有即时生效模式更复杂。

## Migration Plan
不需要数据迁移。已有消息和标签目录保持兼容。
