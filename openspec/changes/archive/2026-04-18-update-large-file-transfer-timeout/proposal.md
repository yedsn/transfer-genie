# Change: 优化大文件传输超时

## Why
当前大文件上传路径存在两个会相互放大的问题：

- WebDAV 上传请求默认带有固定 30 秒总超时，慢速链路或大文件传输即使仍在持续发送字节，也会被提前中断。
- 桌面端 `send_file` 与本机 HTTP 文件入口都会先把整文件读入内存，再进入上传流程，导致大文件场景的启动延迟更高、内存占用更大，也更容易触发超时。

这会让用户在发送大文件时遇到“明明还在传，但仍然报超时”的问题，也会让通过本机 HTTP API 接入的自动化场景在大文件下表现不稳定。

## What Changes
- 将 WebDAV 上传请求与元数据请求分离，移除大文件上传路径上的固定 30 秒总时长限制。
- 为文件发送引入基于文件/临时文件的流式上传路径，避免在发送前把整文件一次性读入内存。
- 让桌面端文件发送与本机 HTTP 文件发送复用同一套长时上传策略、进度回调与错误处理。
- 保留现有上传进度与上传历史行为，但在上传失败时提供更明确的失败原因，并避免将失败传输记为成功。
- 增加覆盖“持续传输超过 30 秒”的回归验证，防止固定请求超时再次回到上传路径。

## Impact
- Affected specs: `message-feed`
- Affected code: `src/webdav.rs`, `src/main.rs`, `frontend/main.js`, `tests/test_local_http_api.js`
