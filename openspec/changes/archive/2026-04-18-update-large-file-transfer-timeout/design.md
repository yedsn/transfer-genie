# 大文件传输超时优化设计

## Context
当前上传实现的关键约束如下：

- `src/webdav.rs` 中的通用认证辅助函数会为 WebDAV 请求附加固定 30 秒超时，这对列表、元数据读取这类短请求是合理的，但对大文件上传不适合。
- `src/main.rs` 中的桌面端 `send_file` 先执行 `fs::read`，本机 HTTP 文件发送入口先执行 `field.bytes()`，都会把完整文件一次性读进内存。
- 现有前端已经具备上传进度展示与上传历史列表，因此本次更适合在后端传输层做收敛式修复，而不是另起一套新交互。

## Goals / Non-Goals
- Goals:
  - 让大文件上传在持续有数据传输时，不会仅因超过固定 30 秒而失败。
  - 降低桌面端文件发送和本机 HTTP 文件发送的大文件内存占用。
  - 保持现有上传进度、发送结果、上传历史的对外行为基本稳定。
- Non-Goals:
  - 不在本次变更中引入 WebDAV 断点续传协议扩展。
  - 不引入后台上传队列、多文件批处理或新的传输管理 UI。
  - 不新增用户可配置的“上传超时秒数”设置。

## Decisions
- Decision: 将 WebDAV 请求策略拆分为“短请求”和“流式上传请求”两类。
  - 短请求继续使用现有的连接/请求保护，适用于 PROPFIND、GET 小文件、DELETE、history 更新等元数据操作。
  - 流式上传请求保留客户端级别的连接超时，但不再施加固定 30 秒总时长限制，避免大文件在持续传输中被统一超时打断。
  - Alternatives considered: 直接把所有 WebDAV 请求都改成无超时；该方案会让真正卡死的短请求更难发现，因此不采用。

- Decision: 为上传路径引入统一的流式源抽象，而不是继续以 `Vec<u8>` 作为主通道。
  - 桌面端文件发送从本地文件句柄分块读取并上送，保留 `Content-Length` 与进度回调。
  - 本机 HTTP 文件发送先将 multipart 文件内容写入临时文件，再复用同一条 WebDAV 上传管线，避免 HTTP 接口先把整文件驻留在内存里。
  - Alternatives considered: 继续沿用 `Vec<u8>`，只取消 30 秒超时；该方案能缓解一部分问题，但大文件的高内存占用和启动延迟仍然存在，因此不采用。

- Decision: 上传成功前不写入成功态历史，不生成“已发送成功”的最终消息状态。
  - 进度事件仍在上传过程中持续发送。
  - 只有远端上传完成后，才写入本地缓存、消息索引与上传历史成功态。
  - 上传失败时保留错误信息，但不得把失败任务显示为成功发送。

- Decision: 缩略图上传、恢复流程等复用 WebDAV 上传辅助能力时，同样不得重新引入固定 30 秒总超时。
  - 本次不要求一次性把所有上传调用都改造成同一实现，但新老辅助函数的超时语义必须统一。

## Risks / Trade-offs
- 临时文件路径需要可靠清理；失败、中断或用户取消后若清理不完整，会留下磁盘垃圾。
- 某些 WebDAV 服务端对流式 PUT 的兼容性较敏感，因此实现必须继续显式提供 `Content-Length`。
- 取消固定总时长超时后，真正“卡住但连接未断开”的上传可能等待更久；这属于可接受权衡，后续如需要可再补 idle timeout。

## Migration Plan
1. 先拆分 `src/webdav.rs` 的请求辅助方法，确保上传路径不再共享固定 30 秒总超时。
2. 引入文件/临时文件驱动的上传辅助能力，并逐步接入桌面端与本机 HTTP 文件发送。
3. 保持前端事件结构不变，校准错误文案与上传历史写入时机。
4. 补充回归测试，覆盖“传输时长超过 30 秒但持续有进度”的场景。

## Open Questions
- 是否需要在后续版本中新增“长时间无进度即失败”的 idle timeout，并暴露为内部常量或可配置项？
