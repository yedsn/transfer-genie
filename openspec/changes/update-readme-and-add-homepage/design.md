## Context

Transfer Genie 已经有可用的桌面应用和若干配套文档，但当前仓库入口依然更偏“文档说明”，而不是“产品展示”。现有 `README.md` 虽然覆盖了很多功能点，但没有在很短时间内清楚传达产品价值、适合的使用场景，以及新访客下一步最应该做什么。与此同时，项目也还没有正式官网首页，这让它在脱离代码仓库语境时不够容易传播。

这次变更会同时涉及仓库内容、静态网页资源和部署说明，但整体实现仍应保持轻量。现有的 `frontend/` 是 Tauri 桌面应用的一部分，不应与面向公众的官网页面混在一起。因此需要一种简单、干净的方案，在不引入大型前端框架、也不打乱桌面端结构的前提下，增加一个更有产品感的首页。

## Goals / Non-Goals

**Goals:**
- 将 `README.md` 升级为更强的项目门面文档，具备清晰层次、精炼价值表达，以及快速进入安装、配置和文档的路径。
- 增加一个响应式单页官网，与 README 的核心叙事保持一致，并给访问者明确的下一步操作入口。
- 保持官网实现轻量、易于静态托管，并与 Tauri 应用前端隔离。
- 尽量复用现有品牌与资源，同时明确还需要补充哪些视觉素材。
- 保证 README 和官网首页内容长期保持一致，避免后续更新后互相矛盾。

**Non-Goals:**
- 不在这次变更中建设多页文档站、博客或 CMS。
- 不重构桌面应用 UI，也不把现有应用前端迁移到新框架。
- 不在这次变更中加入中英双语站点生成能力。
- 不做 README 到官网的自动生成系统，这次只需要一套可维护的手工内容模型。

## Decisions

### 1. 在桌面应用前端之外建立独立静态官网目录

- Decision: 将官网首页放在独立的静态站点目录中，例如 `site/`，而不是复用 Tauri 应用的 `frontend/`。
- Rationale: 官网首页和桌面应用界面解决的是两类不同问题。分离目录可以避免资源和配置耦合，简化部署，也能防止官网相关改动影响桌面应用打包。
- Alternatives considered:
  - 复用 `frontend/`：会把产品 UI 资源和官网内容混在一起，后续维护复杂。
  - 引入 Vite/Next.js：工具更丰富，但对一个单页响应式官网来说复杂度过高。

### 2. 采用 README 与官网共享的内容优先结构

- Decision: 为 README 和官网定义一套统一的信息结构，包括 hero/价值主张、核心功能、主要工作流、快速开始/下载路径、文档入口和项目链接。
- Rationale: 用户明确希望参考 `cc-haha` 来升级 README，因此官网也应该延续同一套产品叙事，而不是变成独立的设计稿。
- Alternatives considered:
  - README 和官网分别独立编写：短期更快，但长期非常容易内容漂移。
  - 直接从 Markdown 自动生成站点：未来可考虑，但对首个官网版本来说属于过度设计。

### 3. 优先采用零构建或最小构建的静态资源方案

- Decision: 官网应能通过任意静态服务器本地预览，并可直接部署到 GitHub Pages、Gitee Pages 或其他基础静态托管平台，无需运行时依赖。
- Rationale: 当前仓库没有专门的 Web 技术栈，最简单的方案维护成本最低。
- Alternatives considered:
  - 服务端渲染方案：对单页官网没有必要。
  - 大量 JS 动效或类应用交互：会提升复杂度，却不显著改善首次了解产品的体验。

### 4. 复用现有品牌资源，并补充少量关键视觉素材

- Decision: 优先复用现有应用图标和已有资源，再补充首页真正需要的最小视觉集合，例如 hero 展示图、截图拼图或工作流示意图。
- Rationale: 一个更像“产品”的首页必须有视觉锚点，但当前项目并不需要立即建立完整设计系统或复杂插画体系。
- Alternatives considered:
  - 纯文字首页：实现最简单，但达不到这次升级的目标。
  - 大量定制图形：视觉更丰富，但首版成本过高。

### 5. 官网源码与发布说明一起交付

- Decision: 在同一次变更中补上官网本地预览和静态发布说明，包括目录约定和相对路径约束。
- Rationale: 一个只能本地存在、无法发布的官网是不完整的，因此需要从一开始就明确部署路径。
- Alternatives considered:
  - 将发布说明延后：会导致这项能力只完成了一半。

## Risks / Trade-offs

- [README 与官网后续内容漂移] -> 保持两者的信息架构平行，并明确哪些章节必须同步更新。
- [官网资源过重] -> 优先使用压缩截图、SVG 图标和单页结构，控制媒体体积。
- [官网实现侵入桌面应用结构] -> 保持站点文件位于独立目录，避免改动 Tauri 运行时路径。
- [暂时没有高质量截图] -> 首版允许采用现有图标配合一到两个精选视觉模块，而不是等待完整截图体系。
- [静态托管目标尚未确定] -> 采用相对路径和宿主无关的目录结构，使同一份产物可用于 GitHub Pages 或 Gitee Pages。

## Migration Plan

1. 盘点现有 README 章节、已有文档、项目链接、图标和可复用的产品视觉素材。
2. 按新的 landing page 信息架构重写 `README.md`。
3. 创建独立官网目录，并补齐首版单页首页所需的 HTML/CSS/资源。
4. 将官网中的链接接到仓库文档、发布页和项目对外入口。
5. 增加本地预览与静态部署说明。
6. 通过静态服务器在本地验证，并检查桌面端与移动端布局表现。
7. 回滚路径：如果官网还不适合发布，可先移除独立站点目录，仅保留 README 升级成果。
